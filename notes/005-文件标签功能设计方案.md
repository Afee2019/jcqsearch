# jcqsearch — 文件标签功能设计方案

## 一、为什么要做标签

### 1.1 目录层级的局限

传统文件系统是**严格的树形结构**——一个文件只能放在一个目录下。但现实中，一个文件往往有多个维度的归属：

- `20260204-数据治理方案.pdf` → 既属于「数据治理」项目，又属于「演示文稿」类型，还属于「2026-Q1」时间段
- 你要把它放在 `项目/数据治理/` 还是 `演示文稿/2026/`？无论放哪，另一个维度就丢了

标签解决的核心问题：**让一个文件可以同时属于多个分类，打破目录层级的单维度限制。**

### 1.2 与现有搜索的互补关系

jcqsearch 目前支持：文件名模糊搜索 + 类型 + 时间 + 大小。

这些都是**文件自身的物理属性**。标签引入的是**人赋予的语义属性**——「重要」「待处理」「客户A」「归档」这些信息不存在于文件名或元数据中，只存在于人的认知里。

标签 + 搜索组合后的能力：

```bash
jcqsearch find 报告 -t pdf -g 重要          # 关键词 + 类型 + 标签
jcqsearch find -g "project:数据治理" --recent 30d  # 标签 + 时间
jcqsearch find -g 待处理 --files-only       # 标签 + 筛选
```

## 二、设计原则

| 原则 | 含义 | 指导 |
|------|------|------|
| **低摩擦** | 打标签必须极其简单，否则没人用 | 一条命令完成，支持批量操作 |
| **渐进式** | 从最简单的场景开始，逐步增加复杂度 | 先平面标签，再规则引擎 |
| **与搜索深度整合** | 标签不是独立功能，是搜索的增强 | find 命令原生支持标签筛选 |
| **标签是索引的一部分** | 标签存数据库，不修改文件本身 | 不改文件名，不写 xattr |
| **容错** | 文件移动/重命名后标签不应默默丢失 | 提供修复机制 |
| **标签质量** | 防止标签泛滥和碎片化 | 提供规范化、合并、清理工具 |

## 三、数据模型设计

### 3.1 方案选型：Toxi 三表模式

业界三种经典标签 schema：

| 模式 | 结构 | 优点 | 缺点 |
|------|------|------|------|
| MySQLicious | 单表，tags 存逗号分隔字符串 | 最简单 | 无法索引，无法全局重命名 |
| Scuttle | 两表，每条 file-tag 关系存一行含标签字符串 | 较简单 | 标签无去重，重命名需全表更新 |
| **Toxi** | 三表（tags + file_tags 关联表） | 全面规范化，标签可独立管理 | 稍复杂 |

**选择 Toxi 模式**——三表完全规范化。理由：

- 标签重命名只需更新一条 `tags` 记录，自动全局生效
- 标签合并、删除都是一等操作
- 标签可以附加属性（颜色、分组、描述）
- 查询性能有保障（PostgreSQL GIN 索引）

### 3.2 数据库表设计

#### 3.2.1 tags — 标签定义表

```sql
CREATE TABLE tags (
    id          SERIAL PRIMARY KEY,
    name        TEXT NOT NULL,                -- 标签名（如 "重要"、"project:数据治理"）
    color       TEXT DEFAULT '',              -- 显示颜色（如 "red"、"#FF5733"、""）
    description TEXT DEFAULT '',              -- 标签说明
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT uq_tags_name UNIQUE (name)
);

-- 标签名模糊搜索（输入时自动补全）
CREATE INDEX idx_tags_name_trgm ON tags USING GIN (name gin_trgm_ops);

COMMENT ON TABLE tags IS '标签定义表';
COMMENT ON COLUMN tags.name IS '标签名，全局唯一，大小写敏感';
COMMENT ON COLUMN tags.color IS '终端显示颜色，用于搜索结果中标签的着色';
```

**标签名规范：**

- 大小写敏感：`Work` 和 `work` 是不同标签
- 允许中文：`重要`、`待处理`、`归档`
- 命名空间约定：用冒号分隔，如 `project:数据治理`、`status:进行中`
- 键值对约定：用等号表示，如 `year=2026`、`author=张三`
- 禁止字符：逗号（用作 CLI 分隔符）、空格（用引号包裹时除外）

#### 3.2.2 file_tags — 文件-标签关联表

```sql
CREATE TABLE file_tags (
    file_id     BIGINT NOT NULL REFERENCES files(id) ON DELETE CASCADE,
    tag_id      INT NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    tagged_at   TIMESTAMPTZ NOT NULL DEFAULT now(),  -- 打标签的时间
    source      TEXT NOT NULL DEFAULT 'manual',      -- 来源：manual / rule / import

    PRIMARY KEY (file_id, tag_id)
);

-- 按标签查文件（最常用的查询方向）
CREATE INDEX idx_file_tags_tag_id ON file_tags (tag_id);

-- 按来源查询（如查看哪些是自动打的标签）
CREATE INDEX idx_file_tags_source ON file_tags (source);

COMMENT ON TABLE file_tags IS '文件-标签关联表';
COMMENT ON COLUMN file_tags.source IS '标签来源：manual=手动, rule=自动规则, import=导入';
```

#### 3.2.3 files 表扩展

在现有 `files` 表中增加指纹字段，用于文件移动后的标签修复：

```sql
ALTER TABLE files ADD COLUMN partial_hash TEXT DEFAULT '';

CREATE INDEX idx_files_identity ON files (size, partial_hash)
    WHERE partial_hash <> '';

COMMENT ON COLUMN files.partial_hash IS '文件指纹（前 8KB 的 SHA-256），用于移动后重新关联标签';
```

**partial_hash 设计说明：**

| 决策 | 选择 | 理由 |
|------|------|------|
| 哈希范围 | 前 8KB | 平衡性能与区分度，5 万文件 < 1 秒 |
| 哈希算法 | SHA-256 | 碰撞概率可忽略 |
| 计算时机 | 仅在打标签时计算 | 不拖慢扫描速度，只有需要追踪的文件才算 |
| 目录处理 | 目录不计算 hash | 目录没有内容可哈希 |

### 3.3 ER 关系图

```
files (已有)                  file_tags                tags
─────────────                ──────────              ──────
id (PK)        ◄──────────── file_id (FK, PK)        id (PK)
path                         tag_id  (FK, PK) ──────► name (UNIQUE)
name                         tagged_at               color
size                         source                  description
partial_hash (新增)                                   created_at
...
```

## 四、CLI 命令设计

### 4.1 命令总览

```bash
# ── 打标签 ──
jcqsearch tag <path> <tags...>              # 给文件打标签
jcqsearch tag <path>                        # 查看文件的标签
jcqsearch untag <path> <tags...>            # 移除标签
jcqsearch untag <path> --all               # 移除所有标签

# ── 批量打标签（基于搜索结果）──
jcqsearch tag --query "报告 -t pdf" work    # 给搜索结果批量打标签
jcqsearch tag -d ~/Desktop -t pptx 演示文稿   # 给指定目录下的 pptx 文件批量打标签

# ── 搜索中使用标签 ──
jcqsearch find 报告 -g 重要                  # 关键词 + 标签筛选
jcqsearch find -g work,project:A           # 多标签 AND 筛选
jcqsearch find -g 重要 -t pdf --recent 7d  # 标签 + 类型 + 时间

# ── 标签管理 ──
jcqsearch tags                              # 列出所有标签及文件数
jcqsearch tags rename <old> <new>           # 全局重命名
jcqsearch tags merge <source> <target>      # 合并标签
jcqsearch tags delete <tag>                # 删除标签（从所有文件移除）
jcqsearch tags clean                       # 清理无文件关联的空标签

# ── 自动标签 ──
jcqsearch autotag                           # 按规则自动打标签
jcqsearch autotag --dry-run                # 预览自动标签结果

# ── 标签修复 ──
jcqsearch tag repair                       # 修复移动/重命名后丢失的标签
```

### 4.2 tag 命令详细设计

#### 4.2.1 给文件打标签

```bash
jcqsearch tag ~/Desktop/报告.pdf 重要 work
```

**执行流程：**

1. 解析路径为绝对路径
2. 在 `files` 表中查找该文件（必须已扫描入库）
3. 如果文件的 `partial_hash` 为空，计算并更新
4. 对每个标签名：若 `tags` 表中不存在则自动创建
5. 插入 `file_tags` 关联（`ON CONFLICT DO NOTHING` 避免重复）
6. 输出结果

**输出示例：**

```
已标记: 报告.pdf
  + 重要  + work
  当前标签: 重要, work, project:数据治理
```

#### 4.2.2 查看文件标签

```bash
jcqsearch tag ~/Desktop/报告.pdf
```

**输出示例：**

```
文件: ~/Desktop/报告.pdf (2.1 MiB, 2026-03-15)

  标签:
    重要         (手动, 2026-03-19)
    work        (手动, 2026-03-19)
    project:数据治理  (规则, 2026-03-20)
    document    (规则, 2026-03-20)
```

#### 4.2.3 批量打标签（核心亮点）

这是最强大的用法——**复用搜索能力批量打标签**：

```bash
# 把所有桌面上的 PDF 都标为 document
jcqsearch tag -d ~/Desktop -t pdf document

# 把搜索结果批量打标签
jcqsearch tag --query "数据治理" project:数据治理

# 把最近 7 天的 pptx 标为 演示文稿
jcqsearch tag --recent 7d -t pptx 演示文稿

# 给大文件标记
jcqsearch tag --larger 500M large-file
```

**执行流程：**

1. 使用与 `find` 相同的搜索参数（`--query`, `-t`, `-d`, `--recent` 等）
2. 执行搜索，获取匹配文件列表
3. 显示匹配数量，请求用户确认
4. 批量打标签
5. 输出汇总

**输出示例：**

```
$ jcqsearch tag -d ~/Desktop -t pdf document

  将为 113 个文件添加标签: document

  确认? [Y/n] y

  ✓ 已标记 113 个文件  (12ms)
```

**设计要点：**

- `tag` 命令复用 `find` 的所有筛选参数，确保用户可以先 `find` 预览，再 `tag` 执行
- 批量操作前展示数量并请求确认（`--yes` / `-y` 跳过确认）
- `--query` 接受与 `find` 相同的关键词语法

### 4.3 find 命令标签集成

#### 4.3.1 基本标签筛选 (`-g` / `--tag`)

```bash
# 搜索带有「重要」标签的文件
jcqsearch find -g 重要

# 多标签 AND 筛选（逗号分隔 = 同时具有所有标签）
jcqsearch find -g 重要,work

# 关键词 + 标签
jcqsearch find 报告 -g 重要 -t pdf
```

**为什么用 `-g` 不用 `-t`：**

`-t` 已被 `--type` 使用。`-g` 取自 "taG" 的尾字母，简短且不冲突。

#### 4.3.2 输出中展示标签

搜索结果增加标签列：

```
  #  名称                          大小       修改时间            标签                 路径

  1   数据治理方案V3.pdf             2.1 MiB   2026-03-15 10:30  重要 work            ~/Desktop
  2   2024-数据治理报告.pdf           856 KiB   2024-02-20 14:22  project:数据治理     ~/Documents

  共 2 条结果（耗时 8ms）
```

标签可以根据 `tags.color` 着色显示。

#### 4.4.1 列出所有标签


3. 删除源标签 `important`

```bash
# 删除无文件关联的空标签
jcqsearch tags clean
```

## 五、自动标签规则

### 5.1 规则定义

在 `config.yaml` 中配置：

```yaml
auto_tag:
  rules:
    # ── 按扩展名 ──
    - match:
        ext: [pdf, docx, doc, odt, rtf]
      tags: [document]

    - match:
        ext: [pptx, ppt, key]
      tags: [演示文稿]

    - match:
        ext: [jpg, jpeg, png, gif, webp, svg, bmp]
      tags: [image]

    - match:
        ext: [mp4, mkv, avi, flv, mov, wmv]
      tags: [video]

    - match:
        ext: [mp3, flac, wav, aac, ogg]
      tags: [audio]

    - match:
        ext: [go, py, js, ts, rs, java, c, cpp, rb, sh]
      tags: [code]

    - match:
        ext: [xlsx, xls, csv]
      tags: [spreadsheet]

    # ── 按目录路径（前缀匹配）──
    - match:
        dir: ~/dev/jcmdm
      tags: [project:jcmdm]

    - match:
        dir: ~/dev/jcqsearch
      tags: [project:jcqsearch]

    # ── 按文件名模式（正则）──
    - match:
        name: "^\\d{8}-.*"      # 以日期开头的文件，如 20260319-xxx
      tags: [dated]

    # ── 按文件大小 ──
    - match:
        larger: 100M
      tags: [large-file]

    # ── 组合条件（所有条件 AND）──
    - match:
        ext: [pdf]
        dir: ~/Documents
      tags: [document, personal]
```

### 5.2 规则匹配逻辑

```
对每个文件:
  对每条规则:
    如果所有 match 条件都满足:
      添加 tags 中的标签（source = "rule"）
```

**条件类型：**

| 条件 | 含义 | 示例 |
|------|------|------|
| `ext` | 扩展名列表（OR） | `[pdf, docx]` |
| `dir` | 目录路径前缀 | `~/dev/jcmdm` |
| `name` | 文件名正则匹配 | `"^IMG_\\d+"` |
| `larger` | 最小文件大小 | `100M` |
| `smaller` | 最大文件大小 | `1G` |

同一规则内多个条件为 **AND** 关系。

### 5.3 执行方式

```bash
# 预览：显示哪些文件会被打哪些标签，但不实际执行
jcqsearch autotag --dry-run

# 执行：对所有已索引文件应用规则
jcqsearch autotag

# 仅对指定路径下的文件应用
jcqsearch autotag -d ~/Desktop
```

**输出示例（dry-run）：**

```
$ jcqsearch autotag --dry-run

  规则                               匹配文件数    标签
  ───────────────────────────────  ──────────  ──────
  ext: [pdf, docx, doc, odt, rtf]       162   document
  ext: [pptx, ppt, key]                  83   演示文稿
  ext: [jpg, jpeg, png, ...]             56   image
  ext: [go, py, js, ts, ...]            245   code
  dir: ~/dev/jcmdm                      892   project:jcmdm
  larger: 100M                           12   large-file

  将为 1,203 个文件添加 1,450 个标签关联
  (实际执行请去掉 --dry-run)
```

### 5.4 与 scan 的协同

自动标签规则**不自动在 scan 时执行**。理由：

- scan 的职责是同步文件系统状态，应保持快速
- 自动标签是语义层操作，用户应显式控制
- 用户可以选择 `scan && autotag` 组合使用

## 六、标签修复机制

### 6.1 问题场景

文件被移动或重命名后，数据库中的 `files.path` 已过时。下次 `scan` 会：

- 在旧路径创建一条"已删除"记录（被清理掉）
- 在新路径创建一条新记录（无标签）

标签关联丢失了。

### 6.2 修复流程

```bash
jcqsearch tag repair
```

**执行流程：**

```
1. 查找"孤儿标签"：有 file_tags 关联但 file_id 对应的 files 记录已不存在的情况
   （文件被删除或路径变化后被清理）

2. 对每个孤儿：
   a. 通过保存的 (size, partial_hash) 在当前 files 表中搜索匹配
   b. 如果找到唯一匹配 → 自动重新关联
   c. 如果找到多个匹配 → 列出候选，提示用户选择
   d. 如果未找到 → 标记为"文件已丢失"

3. 输出修复报告
```

### 6.3 预防措施

为避免标签丢失，scan 的清理逻辑需要调整：

**当前逻辑：** 扫描后直接 `DELETE FROM files WHERE scanned_at < 扫描时间`

**调整为：** 如果文件有标签关联，不立即删除，而是标记为"未找到"状态，保留 N 天供 repair 使用。

```sql
-- 清理时，无标签的文件直接删除
DELETE FROM files f
WHERE f.scan_path_id = $1
  AND f.scanned_at < $2
  AND NOT EXISTS (SELECT 1 FROM file_tags ft WHERE ft.file_id = f.id);

-- 有标签的文件仅标记（后续 repair 处理）
-- 需要给 files 表增加 missing_since 字段
UPDATE files f
SET missing_since = now()
WHERE f.scan_path_id = $1
  AND f.scanned_at < $2
  AND EXISTS (SELECT 1 FROM file_tags ft WHERE ft.file_id = f.id);
```

### 6.4 files 表额外字段

```sql
ALTER TABLE files ADD COLUMN missing_since TIMESTAMPTZ DEFAULT NULL;

COMMENT ON COLUMN files.missing_since IS '文件在文件系统中未找到的时间，NULL=正常存在';
```

## 七、搜索查询实现

### 7.1 标签筛选 SQL

```sql
-- 单标签筛选：查找带有「重要」标签的文件
SELECT f.path, f.name, f.size, f.mod_time
FROM files f
JOIN file_tags ft ON ft.file_id = f.id
JOIN tags t ON t.id = ft.tag_id
WHERE t.name = '重要'
ORDER BY f.mod_time DESC
LIMIT 20;

-- 多标签 AND 筛选：同时具有「重要」和「work」标签
SELECT f.path, f.name, f.size, f.mod_time
FROM files f
WHERE f.id IN (
    SELECT ft.file_id
    FROM file_tags ft
    JOIN tags t ON t.id = ft.tag_id
    WHERE t.name IN ('重要', 'work')
    GROUP BY ft.file_id
    HAVING COUNT(DISTINCT t.id) = 2
)
ORDER BY f.mod_time DESC
LIMIT 20;

-- 关键词 + 标签组合
SELECT f.path, f.name, f.size, f.mod_time
FROM files f
WHERE f.stem % '报告'
  AND f.id IN (
    SELECT ft.file_id FROM file_tags ft
    JOIN tags t ON t.id = ft.tag_id
    WHERE t.name = '重要'
  )
ORDER BY similarity(f.stem, '报告') DESC, f.mod_time DESC
LIMIT 20;
```

### 7.2 查询文件的所有标签

```sql
-- 为搜索结果附加标签信息
SELECT f.id, array_agg(t.name ORDER BY t.name) AS tags
FROM file_tags ft
JOIN tags t ON t.id = ft.tag_id
WHERE ft.file_id = ANY($1)  -- $1 = 搜索结果的 file_id 数组
GROUP BY ft.file_id;
```

## 八、实施路线

### Phase T1：基础标签（MVP）

**目标：** 手动打标签 + 按标签搜索。最小可用版本。

- [ ] 数据库迁移：创建 `tags` + `file_tags` 表，`files` 表增加 `partial_hash`
- [ ] `tag <path> <tags...>` — 打标签 + 自动创建标签
- [ ] `tag <path>` — 查看文件标签
- [ ] `untag <path> <tags...>` — 移除标签
- [ ] `find -g <tags>` — 搜索结果按标签筛选
- [ ] `find` 输出中展示标签列
- [ ] `tags` — 列出所有标签

### Phase T2：批量操作 + 标签管理

## 九、关键设计决策总结

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 存储方案 | PostgreSQL（Toxi 三表） | 规范化，标签可独立管理，性能有保障 |
| 标签格式 | 平面 + 命名空间约定 | 简单灵活，`:` 分隔命名空间，无需多表 |
| 文件身份 | path（主） + partial_hash（辅） | 日常靠路径，移动后靠指纹修复 |
| 标签修改文件？ | 不修改 | 不改文件名，不写 xattr，零侵入 |
| 自动标签时机 | 显式命令，不绑定 scan | 职责分离，用户可控 |
| 批量打标签 | 复用搜索参数 | 用户先 find 确认，再 tag 执行，安全直观 |
| CLI flag | `-g` 表示 tag 筛选 | `-t` 已被占用，`-g` 取自 taG 尾字母 |
| 标签来源追踪 | `file_tags.source` 字段 | 区分手动 / 规则 / 导入，便于审计和清理 |
| 键值对标签 | 用命名约定 `key=value` | 不加表，用 `=` 分隔，搜索时精确匹配 |

## 十、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 标签泛滥 | 标签太多失去意义 | `tags clean` 清理空标签；`tags` 列表展示文件数帮助评估 |
| 标签碎片化 | 同义标签多个版本（大小写、简繁） | `tags merge` 合并；未来可加同义词建议 |
| 文件移动丢标签 | 标签关联失效 | partial_hash + repair 机制 |
| 自动标签误打 | 规则匹配不当 | `--dry-run` 预览；`source` 字段区分来源，可按来源清理 |
| 批量操作误操作 | 大量文件被错误标记 | 批量操作前显示数量并确认；可按 `tagged_at` 时间回滚 |
| 性能：多标签查询慢 | 多标签 AND 需要子查询 | GIN 索引 + HAVING COUNT 模式，小数据量下足够 |
